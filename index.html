Phase 1 Findings

Inferences Derived from Literature Survey

Key design and engineering decisions that emerged directly from studying the 6 scholarly sources.

🔒 Inference 1 — RBAC at Database Level (from Paper 01 & 04)

Sandhu et al. established that role-based access control is most secure when enforced at the data layer, not just the UI. This directly motivated our use of Supabase Row-Level Security (RLS) policies on every table — not just frontend routing guards.

📲 Inference 2 — WebSocket for Real-Time (from Paper 02)

Pimentel & Nickerson demonstrated that WebSocket-based push is significantly more efficient than polling for real-time updates (up to 5× less bandwidth). This justified using Supabase Realtime channels instead of setInterval polling for approval status notifications.

✅ Inference 3 — State Machine for Approval Workflow (from Paper 03)

van der Aalst's BPM research established that deterministic state machines prevent illegal state transitions in workflow systems. The Pending → Under Review → Approved/Rejected enum-driven model prevents double-approval or rollback exploits in our system.

📋 Inference 4 — Immutable Audit via DB Trigger (from Paper 06)

Damiani et al. showed that compliance-grade audit trails must be written by the database engine itself, not application code, to prevent tampering. This motivated our PostgreSQL trigger function that auto-writes to audit_logs on every status change — application layer cannot bypass it.

📸 Inference 5 — SPA Architecture for Dashboard UX (from Paper 05)

Aggarwal's React.js study showed SPAs reduce full-page reloads and enable component-level re-rendering, giving 40–60% faster perceived performance for data-heavy dashboards. This confirmed React as the right choice over server-rendered alternatives like PHP or Django templates.

🎯 Overall Inference — Why This Stack Was Chosen

All 6 papers collectively validated the React + Supabase architecture: RBAC enforcement at the DB layer (Papers 1 & 4), WebSocket-based real-time (Paper 2), deterministic workflow state machine (Paper 3), SPA for UI performance (Paper 5), and database-engine-level audit (Paper 6).

Technology Decision

Why Supabase?

A critical Phase 1 technology decision — justified through literature, requirements analysis, and comparative evaluation.

Reasons Chosen

📊 1. Native PostgreSQL Required

Our approval workflow required relational JOINs between users, requests, actions, and audit_logs. Firebase's NoSQL model would require data duplication and complex denormalisation. PostgreSQL handles this natively with ACID guarantees.

🔒 2. Row-Level Security at Storage Layer

Literature Paper 04 (Momjian) confirms RLS at the DB layer is safest. Supabase's native PostgreSQL RLS means even a compromised API or rogue application cannot read unauthorized records — the DB engine itself enforces isolation.

⚡ 3. Realtime Without Extra Services

Supabase Realtime uses PostgreSQL's logical replication to stream database changes as WebSocket events. This eliminated the need for a separate messaging service (e.g. Redis Pub/Sub or AWS SNS).

🚀 4. Fastest Path to Production

Auth, database, realtime, and file storage in one umbrella replaced what would have been 4–6 separate services. This reduced Phase 2 implementation scope by an estimated 3–4 weeks compared to a custom Node.js + Express backend.

Structure Decisions

📁 Feature-Based Folder Organisation

Pages are grouped by role (admin/, requester/, approver/) rather than by file type. This means all code for the Requester flow lives together — easier to develop and review independently.

🔐 Supabase Client Singleton

The lib/supabase.ts file exports a single authenticated Supabase client instance. All API calls and realtime subscriptions flow through here — avoiding multiple client instances that could cause connection leaks.

📚 Custom Hooks Pattern

Business logic is extracted into custom hooks (useAuth, useRequests, useRealtime). Components remain thin UI layers. This separation makes each hook independently unit-testable without rendering UI.

📋 SQL in Version Control

All database schema, RLS policies, and seed data are committed as .sql files in the supabase/ folder. This makes the database fully reproducible and auditable — no schema drift from manual Supabase Studio edits.
